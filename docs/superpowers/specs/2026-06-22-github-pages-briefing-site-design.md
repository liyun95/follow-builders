# GitHub Pages Briefing Site Design

## Context

This project currently works as an agent skill and feed pipeline for "Follow Builders, Not Influencers." It can prepare digest input JSON from centrally published feeds and deliver text to Telegram or email. The desired experience is different: a public website that archives every daily briefing, with Telegram used only as a notification channel.

The target public URL is:

```text
https://liyun95.github.io/follow-builders/
```

The local repository now pushes to `git@github.com:liyun95/follow-builders.git`, with the original project kept as `upstream`.

## Goals

- Publish a public GitHub Pages site from the same repository.
- Generate one dated briefing page per day.
- Keep every daily briefing as an archive.
- Maintain an index page that links to recent issues.
- Use the "Newspaper Issue" layout direction, similar to the project author's demo.
- Use Chinese as the default site and report language.
- Send a short Telegram notification containing the public issue URL.
- Avoid requiring X/Twitter or podcast transcript API keys in this fork.

## Non-Goals

- Do not build a database-backed app in the first version.
- Do not add user accounts, login, personalization, or search in the first version.
- Do not regenerate raw X/podcast/blog feeds in the fork.
- Do not ask the LLM to hand-write final HTML.

## Architecture

The fork will consume the author's central feeds through the existing `scripts/prepare-digest.js` script. That script downloads:

```text
https://raw.githubusercontent.com/zarazhangrui/follow-builders/main/feed-x.json
https://raw.githubusercontent.com/zarazhangrui/follow-builders/main/feed-podcasts.json
https://raw.githubusercontent.com/zarazhangrui/follow-builders/main/feed-blogs.json
```

The new GitHub Actions publisher will run daily:

```text
node scripts/prepare-digest.js
→ node scripts/generate-briefing.js
→ node scripts/render-site.js
→ commit docs/ updates
→ GitHub Pages publishes
→ Telegram sends the issue link
```

The LLM generates structured briefing data. Deterministic rendering code turns that data into HTML. This keeps the site layout stable and makes the generated content easier to validate.

## Components

### `scripts/generate-briefing.js`

Inputs:

- Prepared digest JSON from `prepare-digest.js`
- Prompt files included in the prepared JSON
- Environment variables for the OpenAI-compatible LLM provider

Responsibilities:

- Call the configured OpenAI-compatible API.
- Instruct the model to use only the feed JSON.
- Ask for a structured JSON response, not HTML.
- Filter low-signal social posts.
- Produce sections for X/Twitter, official blogs, and podcasts.
- Preserve original URLs for every included item.
- Write a dated issue JSON file for the renderer.

Expected environment variables:

```text
OPENAI_API_KEY
OPENAI_BASE_URL
OPENAI_MODEL
```

Default model can be `gpt-5.5`, but the workflow should read `OPENAI_MODEL` so the model can be changed without code edits.

### `scripts/render-site.js`

Inputs:

- Dated issue JSON generated by `generate-briefing.js`
- Existing issue files in `docs/issues/`

Responsibilities:

- Render `docs/issues/YYYY-MM-DD.html`.
- Update `docs/index.html` with recent issue links.
- Render consistent page chrome, metadata, and section structure.
- Include `docs/assets/style.css`.
- Keep all generated URLs relative so the site works under `/follow-builders/`.

### `docs/`

GitHub Pages will publish static files from this directory.

Planned structure:

```text
docs/
  index.html
  assets/
    style.css
  issues/
    2026-06-22.html
```

The index page should show the latest issue prominently and list recent issues in reverse chronological order.

### `.github/workflows/publish-site.yml`

Responsibilities:

- Run on a daily schedule and by manual dispatch.
- Schedule the daily run for `07:30 UTC`, which is `15:30 Asia/Shanghai`. The upstream central feed is scheduled around `06:17 UTC`, so this gives it time to publish before the site reads the latest feeds.
- Install Node dependencies in `scripts/`.
- Run the prepare, generate, and render scripts.
- Commit changed `docs/` files back to the fork.
- Send a Telegram notification only when a new issue page is published successfully.

Required GitHub repository secrets:

```text
OPENAI_API_KEY
OPENAI_BASE_URL
OPENAI_MODEL
TELEGRAM_BOT_TOKEN
TELEGRAM_CHAT_ID
```

Telegram secrets are optional for site generation but required for notification.

## Existing Upstream Feed Workflow

The inherited workflow `.github/workflows/generate-feed.yml` runs `scripts/generate-feed.js`, which requires:

```text
X_BEARER_TOKEN
POD2TXT_API_KEY
```

This fork should not depend on those secrets in the first version. The site publisher should consume the central feeds through `prepare-digest.js` instead. The inherited feed workflow should be disabled, renamed, or otherwise prevented from failing in the fork.

## Page Design

Use the Newspaper Issue direction:

- Header with site name, date, issue number or generated timestamp.
- Small stats summary near the top.
- Editor note or lead summary.
- Section order:
  1. X / Twitter dispatches
  2. Official blogs
  3. Podcasts
- Each item includes title/source, concise summary, and original URL.
- A sidebar or secondary block lists recent issues.
- Mobile-first readability, since Telegram links will often be opened on a phone.

The first version should be restrained and content-focused. It should resemble a serious daily briefing rather than a marketing landing page.

The visual style should use the approved "Editorial Paper" direction:

- Warm light paper-like background.
- Editorial/newsletter typography.
- Strong issue header with date and stats.
- Clean dividers rather than heavy cards.
- Mobile-first single-column reading, with archive/navigation blocks moving below the content on small screens.

## Language and Copy

The report should be written in natural Simplified Chinese by default. The site UI should also be Chinese, using labels such as:

- `今日简报`
- `编辑手记`
- `Builder 动态`
- `官方博客`
- `播客精选`
- `往期简报`

Do not translate `builder` as `建造者`; keep `builder` in English because it reads more naturally in this context. Also keep common AI/product terms in English where Chinese tech readers usually expect them, including:

```text
AI
LLM
agent
API
prompt
token
fine-tuning
RAG
builder
```

Names, companies, products, and source titles should remain in English unless a Chinese name is already standard. Summaries should be Chinese, but original source URLs must remain unchanged.

Telegram notifications should also be Chinese, for example:

```text
今日 AI Builders 简报已更新：
https://liyun95.github.io/follow-builders/issues/YYYY-MM-DD.html
```

## Data Contract

The LLM output should be structured JSON with fields similar to:

```json
{
  "date": "2026-06-22",
  "title": "AI Builders 简报",
  "editorNote": "中文编辑手记...",
  "stats": {
    "xBuilders": 12,
    "totalTweets": 24,
    "blogPosts": 3,
    "podcastEpisodes": 1
  },
  "sections": [
    {
      "id": "x",
      "title": "Builder 动态",
      "items": [
        {
          "source": "Thibault Sottiaux",
          "role": "Codex & ChatGPT at OpenAI",
          "headline": "...",
          "summary": "...",
          "urls": ["https://x.com/..."]
        }
      ]
    }
  ]
}
```

The renderer should validate that each included item has at least one URL. Items without URLs should be skipped.

## Error Handling

- If `prepare-digest.js` fails, the workflow should fail visibly.
- If the LLM returns invalid JSON, retry once with a repair prompt or fail the workflow.
- If there is no meaningful content, generate a short "No new briefing today" issue only if the user explicitly wants that later. For the first version, skip publishing empty issues.
- If Telegram notification fails, the site publish should still remain successful; the workflow should report the notification error.
- If no `docs/` changes are produced, the workflow should exit successfully without committing.

## Security

- API keys must only live in GitHub Secrets or local `.env` files.
- Generated HTML must escape content from the LLM and feeds before rendering.
- The workflow should not print secrets.
- The public site should not include raw API responses or hidden configuration.

## Testing

Implementation should include:

- A local dry run that creates one issue page from current feed data.
- A renderer test or fixture test for HTML escaping and required links.
- A GitHub Actions manual dispatch test after secrets are configured.
- A visual check of the generated page on desktop and mobile widths.

## Follow-Up Operations

- Once the GitHub Actions publisher is verified, replace or pause the local Codex daily Telegram automation so the channel does not receive duplicate briefings.
- If the user later wants morning delivery in Asia/Shanghai, either accept that the site may use the previous upstream feed or switch to self-managed feed generation in the fork.
